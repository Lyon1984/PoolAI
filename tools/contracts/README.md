# Contract tooling

This M0 tool validates the authoritative assets in `docs/contracts/` and the literal SQLSTATE `P0001` boundary in `docs/database/`. It never copies those inputs into an editable source tree.

Commands are run through the locked frontend Node workspace:

```bash
pnpm --dir frontend contracts:validate
pnpm --dir frontend contracts:test
pnpm --dir frontend contracts:generate
pnpm --dir frontend contracts:check
CONTRACT_DIFF_BASE=<40-hex-base-sha> tools/local-dev/run-with-toolchain.sh node tools/contracts/cli.mjs compatibility
```

`validate` checks OpenAPI 3.1 structure, local-only and closed references, the supported JSON Schema vocabulary, operation IDs, exact per-surface security schemes, error media types, AJV compilation, embedded examples, and the stable error catalog. `test` additionally validates JSON/SSE fixtures, Responses sequencing, Chat terminal behavior, and the exact 94-entry SQL `P0001` map, then proves key negative cases fail. `compatibility` requires an exact 40-hex Git base SHA, compares request and response schemas in their respective directions, freezes existing operations and their complete response-status sets, components, error semantics, and `contentMediaType`, and byte-freezes every `.sse` fixture already present at that base; new endpoints, optional fields, stable error codes, and fixtures remain additive. `generate` deterministically writes TypeScript DTOs under `frontend/src/api/generated/` and C# DTOs under `src/PoolAI.Contracts/Generated/`. `all --check` performs every validation and fails if any generated output is stale. The quality workflow derives `CONTRACT_DIFF_BASE` from the exact pull-request base or preceding `main` commit and fails closed when that commit is unavailable.

`docs/contracts/compatibility-resets-v1.json` is a strict audit registry for one accepted, pre-external-release transition; it is not an ignore list. Schema version 1 permits exactly the named M1-E1 record controlled by Issue #44. A reset applies only when its exact Git base, raw base/target OpenAPI SHA-256 digests, accepted ADR markers, and complete sorted diagnostic set all match. Unknown or duplicate fields, a second record, wildcards, stale digests, an unused registered diagnostic, an unregistered diagnostic, or another base fail closed. ADRs are opened through the repository's canonical, no-follow, non-blocking regular-file reader. Once the registry exists in a comparison base, its sole accepted record and ADR bytes are immutable. The record remains as transition history after merge but is inert for later bases; normal `/v1` compatibility rules then apply again. Any future reset would require an explicit schema, policy, ADR, and tooling change rather than another registry entry.

`docs/contracts/compatibility-windows-v1.json` is a separate strict registry for the normal `/v1` compatibility-window process; it does not modify or extend the immutable reset registry. Each window binds one full Git base SHA, raw base/head OpenAPI SHA-256 digests, one ADR, one approval-control Issue, and the complete sorted diagnostics. A `proposed` entry must have `approvalEvidence: null`: the compatibility command verifies its exact binding and then fails with `pending approval`, so the candidate is never an exemption. Approval is one atomic governance transition: registry `status: accepted` plus a non-placeholder permanent Issue-comment URL, and matching ADR `Status: Accepted` plus the identical evidence URL. Accepted history is immutable; a proposed-to-accepted history change may touch only those approval markers. Wildcards, unknown keys, duplicate IDs or bases, stale hashes, partial/extra diagnostics, unsafe ADR files, placeholder evidence, changed accepted history, or reuse from another base fail closed. After the protected target merges it becomes the ordinary comparison baseline and the exact window record is inert.

The C# generator uses `JsonElement` only for explicitly approved mixed-shape properties and the `ChatMessage`, `LoginResult`, `ResponseCompletedOutputItem`, `ResponseInProgressOutputItem`, and `ResponseStreamEvent` root unions. New union locations, schema shapes, or keywords fail generation until the mapping is reviewed. Generated DTOs compile as part of `PoolAI.Contracts`; neither language keeps an independently edited copy of the authoritative OpenAPI or error catalog. Generated DTOs preserve transport fields and null/omission semantics, but the AJV and state-machine validators remain authoritative for cross-field, enum, const, and stream-sequencing rules.
