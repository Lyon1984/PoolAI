# PoolAI 开发执行规格 v1.0

> 状态：Release 1 开发基线（冻结）
> 适用实现：.NET 10 / ASP.NET Core 10、EF Core 10 + Npgsql 10、PostgreSQL、Redis、Vue 3
> 最后更新：2026-07-16
> 目标：把 [`系统重构方案-v1.0.md`](系统重构方案-v1.0.md) 收敛为可拆分、可测试、可发布的执行契约。

本文中的“必须”“不得”“应”是验收要求。若实现与本文、OpenAPI、数据库基线发生冲突，先通过 ADR 修改契约并同步测试，不允许以代码现状反向覆盖需求。

## 1. 执行基线与交付边界

### 1.1 版本定义

| 版本 | 定位 | 是否为当前发布门槛 |
|---|---|---|
| R1.0 | 工程、身份、控制面、供应池、Group 配额账本基础 | 是，作为 R1.1 前置里程碑，不单独对外承诺模型能力 |
| R1.1 | OpenAI/Codex 可生产使用的完整垂直切片 | 是，本规格的首个 GA 门槛 |
| R1.2 | Anthropic Messages 增量 | 否；R1.1 验收后独立开发、独立回归和发布 |

“Release 1 完成”默认指 R1.1 通过第 12 节全部发布门。R1.2 不阻塞 R1.1。

### 1.2 R1.1 范围

| 能力域 | R1.1 必须交付 |
|---|---|
| 网关协议 | `POST /v1/responses` 流式与非流式；`POST /v1/chat/completions` 流式与非流式；`GET /v1/models`；`GET /v1/usage` |
| 上游 | 显式 `provider=openai/openai_compatible` 的 API Key Account 及 OpenAI-compatible Base URL；公开 `platform=openai` 仅表示入站协议族；请求转换、SSE 增量透传/解析、错误标准化、逐 attempt 故障切换 |
| 身份 | 闭合开户；管理员创建用户；邮箱密码登录；可选 TOTP；SMTP + 数据库 email outbox 的忘记/重置密码；会话刷新与撤销 |
| 权限 | Admin、Operator、Auditor、User 四角色；后端操作级授权和前端路由守卫 |
| 访问凭据 | 用户 API Key 创建、列出、禁用、启用、撤销；Key 固定绑定一个 Group；原文只显示一次 |
| 订阅 | 无价格 Subscription Template；管理员手工分配、延长、暂停、撤销、恢复；按有效期控制 Group 访问 |
| 供应池 | Group、Channel、Account、Group–Account 绑定、模型映射、优先级、Account 健康、Account 并发 lease、同 Group 内故障切换；R1 无 User lease |
| 配额 | Group 是唯一累计 Token 限制主体；手工设置总量、调整总量、手工重置；预留、核销、释放、过期回收、审计 |
| 用量 | 每个上游 attempt 的 Account Token 事实；Group/Account 小时聚合；订阅用户查看 Group 池总体用量；管理员下钻 |
| 前端 | 登录与安全、API Key、订阅、Group 池用量、`/key-usage`；用户/Group/订阅/Channel/Account/用量/审计管理页 |
| 运维 | 结构化日志、指标、追踪、审计、健康检查、告警、备份恢复、迁移器、email outbox Worker、容器化部署和发布门 |

为支持上述闭环，控制面 API 必须覆盖资源的查询与操作，但其精确路径、Schema、状态码和错误码以 `docs/contracts/openapi-v1.yaml` 与 `docs/contracts/error-catalog.md` 为准。

### 1.3 明确延后

| 延后项 | 最早目标 | R1.1 中的处理 |
|---|---|---|
| Anthropic `/v1/messages`、`/v1/messages/count_tokens` | R1.2 | 不注册公开路由；不得以未完成开关暴露 |
| Gemini、Antigravity、Grok、Bedrock | R2+ | 不创建适配器和管理入口 |
| 图片、视频、Embeddings、批任务 | R2+ | 不注册对应公开路由；在已支持路由中提交未支持的媒体/能力字段时返回 `unsupported_feature`，且不创建余额冻结替代逻辑 |
| Responses WebSocket | R2+ | R1.1 仅 HTTP/SSE |
| 上游 OAuth/setup-token/Service Account | R2+ | R1.1 Account 仅 API Key 凭据 |
| 用户自助注册、第三方登录、OIDC | R2+ | `/register` 不存在；不能只靠前端隐藏 |
| 外部身份/权益同步 | R2+ | R1.1 订阅只允许管理员手工操作 |
| 自动配额周期、定时重置 | R2+ | 不运行调度任务，不接受 cron/reset policy |
| 跨 Group fallback | R2+ 重新评审 | R1.1 严禁；`routing_group_id` 必须等于 `quota_group_id` |
| Go 版本在线迁移、双写、兼容旧数据库 | 独立兼容轨 | 不阻塞绿地 R1.1；需要时另立迁移规格、演练与退出门 |
| 多区域、多活、跨区域配额账本 | R2+ | R1.1 为单 PostgreSQL 权威域 |

### 1.4 永久删除的非目标

以下不是“延后”，而是本项目非盈利边界，禁止进入领域模型、数据库、OpenAPI、菜单、指标或配置：

- 计费、定价、账单、余额、充值、提现、订单、支付、退款、发票、收入、成本、利润；
- 付费套餐、模型单价、倍率、金额窗口、余额冻结；
- 兑换码、优惠码、邀请码奖励、推广、返利、分销；
- User、API Key、Account、平台级累计 Token 配额或个人额度包。

允许保留的“限制”只有 Group 累计 Token 总量，以及安全/容量性质的 RPM、并发 lease、请求体大小、上游限流与健康熔断；这些限制不得被描述为可购买权益。

### 1.5 绿地部署假设

R1.1 从空 PostgreSQL 数据库部署，使用 `PoolAI.Migrator` 创建基线并由一次性命令引导首个 Admin。生产拓扑至少包含 API、Worker、Migrator Job、PostgreSQL 和 Redis；API/Worker 数据库角色没有 DDL 权限。Go 实现只作为行为参考，不要求复用表、密码散列、Token、主键或在线双写。

## 2. 冻结的产品与技术决策

| ID | 决策 | 验收含义 |
|---|---|---|
| DEC-001 | 后端统一使用 `net10.0` 和 ASP.NET Core 10 | 不建立第二套 Go 运行时；CI 固定 SDK 与锁文件 |
| DEC-002 | R1.1 是绿地发布 | 只验证全新迁移和备份恢复；旧 Go 数据迁移另立项目 |
| DEC-003 | 闭合开户 | 只有 Admin 能创建用户；公开注册路由不存在 |
| DEC-004 | 登录为密码 + 可选 TOTP | 密码验证通过且启用 TOTP 时必须完成二次验证才签发会话 |
| DEC-005 | Access JWT 15 分钟，Refresh Token 30 天且单次轮换 | JWT 用注入 `TimeProvider`、NTP 同步节点时钟且 `ClockSkew=30s`；刷新成功后旧 Refresh Token 立即失效；重放撤销整个 family |
| DEC-006 | 角色固定为 Admin、Operator、Auditor、User | 一名用户 R1.1 只有一个角色；权限以服务端 Policy 为准 |
| DEC-007 | 订阅只由 Admin/Operator 手工维护 | 不存在购买、兑换、自助开通或外部同步入口 |
| DEC-008 | 同一 `user_id + group_id` 只有一条 canonical Subscription | 分配、延长、暂停、撤销、恢复更新同一行并追加审计；不得创建多条有效记录后合并 |
| DEC-009 | Group 是唯一累计 Token 配额主体 | 用户、Key、Account 只有 Token 统计维度，没有累计总量或剩余量 |
| DEC-010 | Group 总量必须有限 | `total_tokens` 为 `1..9,007,199,254,740,991`；不支持 unlimited |
| DEC-011 | Token 管理输入与累计事实使用不同数值契约 | `total_tokens`/单 attempt `estimate` 是 `1..2^53-1` JSON number；事实、adjustment 和累计值使用 PostgreSQL `numeric(78,0)`/.NET `BigInteger`，quota/Group/Account 累计对外为最长 78 位十进制 string，不截断；78 位溢出使事务失败并发 P0 告警 |
| DEC-012 | Group 周期只允许手工重置 | R1.1 没有自动周期、`next_reset_at` 或 cron 行为；订阅续期不重置 Group |
| DEC-013 | 预留失败按数值原因精确区分 | `consumed>=total` 为 `group_quota_exhausted`；`estimate>total-consumed` 为不可立即重试的 `group_quota_insufficient`；否则仅因 reserved 不足时为 `group_quota_reserved` + `Retry-After: 1`；`reserved-full` 仅为内部派生标签，公开 quota status 仍只有 active/exhausted/disabled |
| DEC-014 | PostgreSQL 是配额准入与核销真相 | Redis/内存缓存不得决定放行；任何依赖 PostgreSQL 的公开路径在数据库不可用时 fail-closed，并统一返回 `503 dependency_unavailable` |
| DEC-015 | 每个上游调用是独立 attempt | 每次切换先结算旧 attempt，再在同 Group 内选择 Account 并创建新 reservation |
| DEC-016 | R1.1 禁止跨 Group fallback | 所有事实中 `routing_group_id == quota_group_id`；不相等视为数据完整性故障 |
| DEC-017 | ID 由应用生成 UUIDv7 | `request_id` 每次入站唯一；`attempt_id` 全局唯一；同请求 `attempt_index` 从 0 连续递增 |
| DEC-018 | 控制面写操作支持并发与重试 | 配额调整/重置、订阅变更、用户状态、Account 凭据替换与 Group Supply Configuration 写入必须使用 `Idempotency-Key`；版本化资源同时使用自身 `If-Match`，Group 与 Supply Configuration 强 ETag 独立且不能交叉授权 |
| DEC-019 | API Key 只存 HMAC-SHA256 摘要 | HMAC 使用服务端 pepper；原 Key 只在创建响应显示一次，只持久化 prefix、hash 和元数据 |
| DEC-020 | API Key 的 Group 绑定不可修改 | 换 Group 必须创建新 Key 并撤销旧 Key，避免历史归属漂移 |
| DEC-021 | 配额耗尽不阻断查询 | 模型请求返回 `429 group_quota_exhausted`；有效 Key 仍可 `GET /v1/usage` |
| DEC-022 | 计量优先取上游值 | dispatch 后仅在 Adapter 有可靠“未写出任何上游字节/明确未执行拒绝”证据时以 `confirmed_no_execution` settle 0；其他无 usage 的歧义结果保守估算并标记；后得真实值用 adjustment 修正，实际值不得截断 |
| DEC-023 | 网关顺序固定 | 入站鉴权 → PostgreSQL canonical Key/User/Subscription/Group 强读 → 模型映射/估算 → 强读 Supply 的 `group_supply_configurations`、configured Channel、`group_accounts`、Account lifecycle/health → 同 Group Account 调度/lease → attempt reservation（函数再次锁定并验证当前 configured Channel）→ 持久化 dispatch fence并提交 → 上游 → settle；仅 fence 提交前失败可 release，任何上游字节只能在 fence 提交后发送，每个 failover attempt 在调度前重复全部 canonical 强读 |
| DEC-024 | R1.1 只支持 OpenAI 协议族的 API Key 上游凭据 | Account/Channel 创建时显式且不可变的 `provider` 仅为 `openai/openai_compatible`；`platform=openai` 不是 Provider selector；凭据写入 envelope encryption，读取 API 从不返回明文，只允许替换 |
| DEC-025 | `/v1/usage` 是 Group 池视图 | Group 由当前 Key 服务端确定；不接受 group 参数；不返回用户、Key 或 Account 明细 |
| DEC-026 | 时间判定使用明确权威时钟 | Subscription/API Key/一次性 Token/Group reservation 用 PostgreSQL clock；Redis Account lease 与 RPM 用 Redis `TIME`；JWT 用注入 `TimeProvider` + NTP 且 `ClockSkew=30s`；TOTP 30s 步长、接受 ±1 步并持久化 last accepted step 防重放 |
| DEC-027 | 商业功能物理缺席 | 不能以 feature flag、隐藏菜单或空表“暂存”支付、价格、余额、兑换模块 |
| DEC-028 | 撤销后的新准入强读 canonical 状态 | Admin disable/revoke 提交成功后，所有新的模型请求和 failover 都必须从 PostgreSQL 强读 API Key/User/Subscription/Group；Redis 只预筛，不能授权 |
| DEC-029 | GroupQuota 与 Supply 配置分离 | 按 ADR 0001，GroupQuota 独占 Group/version，Supply 独占 `group_supply_configurations` 及其 `group_accounts` 子项；Channel/绑定变化只推进 Supply version，不改变 Group/ETag。`IGroupSupplyReadiness` 向无数据所有权的 `Application.Orchestration` 返回 opaque versioned evidence，GroupQuota 只在自身 UoW 持久化 token/observed_at 并激活，不解析或把 token 当 lease/capability |
| DEC-030 | Supply 状态、健康与退役引用分离 | Account/Channel 持久状态均为 active/disabled/retired；Account 健康另为 unknown/healthy/degraded/cooling/unhealthy；新 attempt/failover 强读，已准入 attempt 仍结算。退役只读 Supply 自有事实：任一 enabled binding 阻止 Account，任一 non-null Supply Configuration 引用阻止 Channel；先 PATCH 清理/禁用，Supply 不查询 Group status |
| DEC-031 | 控制面授权不信任 JWT 内的旧状态 | 每个已认证控制面请求强读 PostgreSQL User status/role/token_version；用户状态或角色变更递增 token_version |
| DEC-032 | Worker 单 owner 使用 PostgreSQL session advisory lock | R1 不实现 Redis Worker leader/fencing key；每类 job 使用专用 NpgsqlConnection 持有版本化 advisory lock，且仍保持数据库幂等 |
| DEC-033 | 忘记密码通过 SMTP + PostgreSQL email outbox | 公开请求不枚举邮箱；一次性 Token 只存 hash；Worker 有界重试、死信与审计，邮件内不含密码 |
| DEC-034 | R1 设置面只读 | 不提供运行时 settings mutation API；配置和 secret 只通过部署流水线变更 |
| DEC-035 | R1 不实现 User 并发 lease | 只保留 Account lease 与 Group RPM 作为容量/安全限制；它们不是用户或 Account 累计 Token 配额 |
| DEC-036 | 非流 reservation 需主动续租且有总截止 | lease 300s、每 60s 续租、非流总时长最多 600s；超时取消上游、有限 drain 并核销或保守估算 |
| DEC-037 | 每个可执行 Host 恰有一个 Composition Root，并执行完整 Clean Architecture 依赖规则 | Api、Worker、Migrator 各自在 `Program.cs` 组装自己的根容器；Application 不依赖 Infrastructure，Endpoint 不绕过用例，任何层不返回 EF Entity/`IQueryable` 作为跨层契约 |
| DEC-038 | 一条 Command 只有一个显式 Unit of Work | 按 ADR 0002，Application Handler 通过 vendor-neutral factory 成为 begin/commit 唯一 owner；`IUnitOfWork` 暴露独立且无 commit/dispose 的 `Context`，不实现该 context；Repository、Audit/Outbox Appender、幂等组件不得自行提交；业务事实、审计、幂等响应和 integration event append 必须参加同一 PostgreSQL 事务，发布只能发生在 commit 后 |
| DEC-039 | attempt 在调用上游前必须持久化 dispatch fence | reservation 从 `not_started` 原子标记为 `started` 后才允许发送上游；只有 `not_started` 可零消耗 release/expire；`started` 后结果不确定时按保守估算核销，真实 usage 迟到后用 adjustment 修正 |
| DEC-040 | `usage_requests/usage_attempts/usage_attempt_adjustments` 是 GroupQuota 拥有的 immutable settlement facts | Usage 不建立指向这些表的可写 Entity/Repository；只消费版本化 Integration Event 或经 GroupQuota 发布的只读事实契约构建详细查询和小时聚合 |
| DEC-041 | 数据面采用 phase-aware resilience pipeline 与进程级 Bulkhead | 数据面、控制面和 `/v1/usage` 使用独立 concurrency policy；每 Account breaker 冻结 closed/open/half-open；重试必须受总截止、attempt 截止、retry budget 和“是否已 dispatch/是否已输出业务事件”共同约束 |
| DEC-042 | 所有 envelope encryption 使用版本化 AEAD 契约 | envelope 固定 schema/algorithm/key-id/nonce/tag/ciphertext/wrapped-DEK，AAD 绑定实体 ID、字段和用途；历史 key resolver、重包裹、未知版本 fail-closed 与备份恢复进入发布门 |

任何改变 DEC-008、DEC-009、DEC-011、DEC-013、DEC-014、DEC-016、DEC-019、DEC-027～032 或 DEC-035～042 的修改均属于架构变更，必须经过 ADR、数据影响分析、威胁建模和完整回归。

### 2.1 Solution 物理结构

R1.1 使用模块化单体，不使用“所有业务都可互相引用”的三层大项目。每个业务模块由 `*.Abstractions` 和实现程序集组成；实现程序集内部再按 Domain/Application/Infrastructure/Endpoints 分目录，并通过 `internal` 隐藏实现。

~~~text
PoolAI.sln
  src/
    PoolAI.Api/                         # HTTP/SSE Host；本进程唯一 Composition Root
    PoolAI.Worker/                      # 后台任务 Host；本进程唯一 Composition Root
    PoolAI.Migrator/                    # DDL/基线迁移 Host；本进程唯一 Composition Root
    PoolAI.Application.Orchestration/   # 无数据所有权；仅跨模块控制面用例
    PoolAI.Contracts/                   # 版本化公开 HTTP/SSE DTO，无业务实现
    PoolAI.BuildingBlocks/              # Result、ID、事务/outbox 抽象等最小公共原语
    PoolAI.Database.Migrations/         # 唯一 Schema/Migration 资产
    PoolAI.Infrastructure.Postgres/     # Host-local Npgsql/UoW/session-lock runtime；无业务 SQL/schema
    Modules/
      PoolAI.Modules.Identity.Abstractions/
      PoolAI.Modules.Identity/
      PoolAI.Modules.SubscriptionAccess.Abstractions/
      PoolAI.Modules.SubscriptionAccess/
      PoolAI.Modules.GroupQuota.Abstractions/
      PoolAI.Modules.GroupQuota/
      PoolAI.Modules.Supply.Abstractions/
      PoolAI.Modules.Supply/
      PoolAI.Modules.Routing.Abstractions/
      PoolAI.Modules.Routing/
      PoolAI.Modules.Usage.Abstractions/
      PoolAI.Modules.Usage/
      PoolAI.Modules.Operations.Abstractions/
      PoolAI.Modules.Operations/
      PoolAI.Modules.Gateway.Abstractions/
      PoolAI.Modules.Gateway/
    Adapters/
      PoolAI.Adapters.OpenAI/
  tests/
    PoolAI.UnitTests/
    PoolAI.ArchitectureTests/
    PoolAI.ContractTests/
    PoolAI.IntegrationTests/
    PoolAI.EndToEndTests/
    PoolAI.LoadTests/
~~~

`PoolAI.Contracts` 只放公开传输契约；模块不能把 EF Entity 当 DTO。`PoolAI.BuildingBlocks` 必须保持小而稳定且 vendor-neutral，禁止放 Npgsql/EF 类型、业务 Service、通用 Repository、God DbContext 或跨模块 DTO。

按 [`ADR 0002`](architecture/adr/0002-introduce-shared-postgres-transaction-runtime.md)，`PoolAI.Infrastructure.Postgres` 是非 bounded-context 的共享技术 Infrastructure。它只承载 Host-local `NpgsqlDataSource`、显式 connection/transaction UoW、concrete non-committing context、专用连接的 PostgreSQL session advisory lock 与无业务规则的配置/诊断；不得包含业务表 SQL、Repository、EF Entity、Migration、Idempotency/Outbox/Audit 语义、quota wrapper 或模块 Worker。业务 SQL 仍在表 owner 的模块 Infrastructure；Schema 与 migration 仍只有 `PoolAI.Database.Migrations`/Migrator 拥有。

### 2.2 模块所有权和公开接口

| 模块 | 独占数据/责任 | 可公开的 Abstractions | 禁止承担 |
|---|---|---|---|
| Identity | User、Role、Password/TOTP、Session/Refresh family、API Key HMAC/状态、password-reset one-time token 与 email outbox | `IIdentityAuthenticator`、`IUserStatusReader`、`IApiKeyAuthenticator`、`IApiKeyIssuer`，只返回最小 ID/状态快照 | Subscription 判断、Group 配额、Account credential、模型转发 |
| SubscriptionAccess | Template、canonical Subscription、有效访问判定、分配/延长/暂停/恢复/revoke | `ISubscriptionAccessReader`、`ISubscriptionCommandService`、有效 access snapshot | 个人 quota、价格、购买、兑换；自行签发未校验 Group 的 Key |
| GroupQuota | Group 生命周期/version、opaque activation evidence、Quota stable row、Period、Reservation、Quota Event、`usage_requests/usage_attempts/usage_attempt_adjustments` immutable settlement facts；dispatch fence 与核销事实同事务 | `IGroupStatusReader`、Group activation command、`IGroupQuotaLedger`、`IReservationHandle`、`IAttemptSettlementFactReader`、quota snapshot、版本化 Integration Event | `group_supply_configurations`/Account/`group_accounts` 写入、Account 调度、解析 Supply token、查询投影/聚合、用户个人额度、跨 Group fallback |
| Supply | Channel、Account、`group_supply_configurations` Aggregate 及其 `group_accounts` 子项/配置命令、模型能力/映射、credential envelope、健康事实 | `ISupplyCatalog`、`IAccountCandidateReader`、`IAccountHealthWriter`、`IModelCatalog`、`IGroupSupplyReadiness`、`IGroupSupplyConfigurationCommands` | 配额扣减、Group 状态/version/activation evidence 变更、协议 DTO、订阅、返回明文 credential |
| Routing | 同 Group 候选过滤、Account 选择、Redis lease、粘性和健康反馈 | `IAccountRouter`、`IAccountLease` | 跨 Group 选择、配额放行、HTTP 协议转换、用量落账 |
| Gateway | 固定 Stage 编排、公开模型路由、取消/流式生命周期、attempt/failover orchestration | `IProtocolAdapter`、`IUpstreamAdapter`、规范化 request/event/error/usage | 直接拥有业务表、直接更新 quota 计数、绕过模块接口查 DbContext |
| Usage | 消费 GroupQuota 版本化事件、经发布事实契约构建详细 query model、小时聚合、Account 报表、`/v1/usage` 组装与对账 | `IPoolUsageReader`、`IAccountUsageReader`、`IUsageAggregationCheckpoint` | 写 settlement fact、同步决定 quota 准入/核销、修改 Group total、按货币换算 |

表所有权不可被“只读 DbContext”绕过；跨模块读取走 Abstractions、版本化 Published Fact/View 或 Integration Event。`PoolAI.Database.Migrations` 可以创建跨表外键/索引；`group_supply_configurations.group_id -> groups.id` 只是存在性/完整性依赖，不转移写 owner。仅登记两个数据库只读例外：quota entry point 为保证准入线性化，可按固定白名单锁定 Identity、SubscriptionAccess、`group_supply_configurations` 与其他 Supply canonical 行，验证请求 Channel 等于 configured Channel，但只能写 GroupQuota settlement facts 并通过当前事务 append Operations 表；Group activation guard 可只读 quota/Supply readiness 阻断绕过 Orchestrator 的明显非法写入，但不锁定/写 Supply，也不形成跨 Context 永久不变量。除这些白名单和 Audit/Idempotency/Outbox/Inbox appender 外，每个模块只能更新自己拥有的表。

PoolAI.BuildingBlocks ──────────> PoolAI.Infrastructure.Postgres（仅 vendor-neutral UoW ports）
PoolAI.Infrastructure.Postgres ─> 各模块 Infrastructure + Api/Worker Composition Root（仅 runtime mechanics）
Operations.Abstractions ────────> 所有模块实现（仅加入当前 UoW 的审计/outbox接口）

Identity.Abstractions ──────────> SubscriptionAccess
GroupQuota.Abstractions ────────> SubscriptionAccess
Supply.Abstractions ────────────> Routing
GroupQuota.Abstractions ────────> Usage

Identity.Abstractions ──────────────┐
SubscriptionAccess.Abstractions ────┤
GroupQuota.Abstractions ────────────┼─> Application.Orchestration
Supply.Abstractions ────────────────┘

Identity.Abstractions ──────────┐
SubscriptionAccess.Abstractions ├─> Gateway
GroupQuota.Abstractions ────────┤

约束：

1. 模块实现只能引用其他模块的 `*.Abstractions`，绝不能引用其他模块实现程序集、内部 Entity、DbContext 或 Endpoint。
2. Abstractions 只能引用 `PoolAI.BuildingBlocks`；它们之间也不得形成环。Domain/Application/Endpoints/Abstractions/Contracts/Orchestration 禁止引用 Npgsql 或 `PoolAI.Infrastructure.Postgres`。
3. `Api` 不写业务规则，只认证 transport、绑定 DTO、调用用例并注册模块；`Worker` 不注册公开 Endpoint；`Migrator` 不引用 Redis、HTTP Adapter、模块运行时或 Secret 解密服务。
4. Gateway 不直接引用 OpenAI Adapter；Api Composition Root 同时注册 Gateway 和 Adapter，依赖反转到 `Gateway.Abstractions`。
5. SubscriptionAccess 的“创建 Key”用例先检查 canonical Subscription/Group，再调用 Identity 的 `IApiKeyIssuer`；Identity 不反向依赖 SubscriptionAccess，因此没有循环。
6. Supply 与 GroupQuota 互不引用；`group_supply_configurations.group_id` 的存在性由数据库 FK 保证，`group_accounts` 只引用该 Supply Aggregate。若 Supply 用例需要 Group 业务状态则由 `PoolAI.Application.Orchestration` 组合，而不是让 Supply 查询 GroupQuota。Group 激活由该 Orchestration 顺序调用 `IGroupSupplyReadiness` 与 GroupQuota command port；readiness 是带 opaque token/数据库 observed_at 的点时证据，不建立跨 Context 事务。GroupQuota 只存储/发布 evidence，不解析它；观察后供应失效不回滚或暗改 Group 状态/version，运行时仍逐 attempt 强读拒绝并告警。
7. `PoolAI.Application.Orchestration` 只依赖模块 Abstractions，不拥有 Entity、Repository、DbContext、Endpoint 或表；它不能成为 Service Locator，也不能绕过拥有者端口直接执行 SQL。
8. 只有模块 `Infrastructure` namespace、技术 Adapter、测试及 Api/Worker Composition Root 可以使用 `PoolAI.Infrastructure.Postgres`。模块实现项目因物理合并逻辑层而携带该引用时，Architecture Test 必须证明 Domain/Application/Endpoints namespace 未使用其类型。共享项目不接受任意 SQL delegate，不提供 Generic Repository/God DbContext，不使用 `TransactionScope`、ambient/`AsyncLocal` transaction lookup 或静态/全局 connection registry。

### 2.4 Host 加载范围与 Adapter 注册

| Host | 必须加载 | 明确不加载 |
|---|---|---|
| Api | Host-local `PoolAI.Infrastructure.Postgres` runtime、Application.Orchestration、Identity、SubscriptionAccess、GroupQuota、Supply、Routing、Usage、Operations、Gateway、OpenAI Adapter | Worker loops、DDL runner、延后平台 Adapter |
| Worker | Host-local `PoolAI.Infrastructure.Postgres` runtime/session lock、Identity email outbox sender、GroupQuota reservation sweeper/outbox、Supply health、Usage aggregator/rebuild、Operations 告警 | Subscription 时间状态写回、HTTP/SSE Endpoints、密码登录、协议 Adapter、DDL runner、Redis leader lease |
| Migrator | Database.Migrations、migration history/checksum、migration-specific session advisory lock、bootstrap-admin 子命令的最小 Identity 写入器 | runtime application UoW、业务 Endpoint、Redis、Gateway、Worker loop、Account secret 读取 |

Api、Worker、Migrator 各自的 `Program.cs` 是该进程唯一 Composition Root；Api/Worker 各自注册独立 data source/UoW runtime，不跨进程共享 `NpgsqlDataSource`、connection、UoW 或根 `IServiceProvider`。Migrator 继续使用 migration-specific 路径，不能因共享 runtime 而把 Schema ownership 交给 Api/Worker。三个 Host 可以复用无业务规则的 `AddIdentityModule()`、`AddGroupQuotaModule()`、`AddOpenAiAdapter()` 注册扩展，但不得引用彼此 Host、共享根 `IServiceProvider` 或使用静态 Service Locator。R1.1 禁止反射扫描目录、运行时下载插件或按字符串动态加载程序集。每个 Adapter 声明唯一的 `(inbound_protocol, upstream_type, operation)` capability，启动时发现重复注册或缺少 R1.1 capability 立即失败。

`IProtocolAdapter/IUpstreamAdapter` 只接收规范化且脱敏的 `AttemptContext`、请求流和 CancellationToken，返回规范化 event/usage/error；Adapter 不得注入 DbContext、Redis、quota ledger、Subscription 或 audit writer。Account credential 由 Supply 以短生命周期受保护句柄提供，不出现在日志/trace/DTO。

### 2.5 Clean Architecture、CQRS 与 Unit of Work

目录不是依赖规则的替代品。每个模块必须满足以下单向矩阵；若 namespace 级 Architecture Test 无法可靠证明，应拆成独立程序集：

| 层 | 可以依赖 | 禁止依赖/返回 |
|---|---|---|
| Domain | 本模块 Domain、最小 `BuildingBlocks` | ASP.NET Core、EF/Npgsql、`PoolAI.Infrastructure.Postgres`、Redis、SMTP、Adapter、Application/Infrastructure/Endpoints |
| Application | Domain、本模块 Ports、其他模块 `*.Abstractions` | Infrastructure、Endpoint、`PoolAI.Infrastructure.Postgres`、具体 DbContext/Npgsql/Redis Client；不得返回 EF Entity、tracked collection 或 `IQueryable` |
| Infrastructure | Application Ports、Domain、EF/Npgsql/`PoolAI.Infrastructure.Postgres`/Redis/SMTP/crypto | Endpoints、其他模块实现；不得把 persistence model/concrete PostgreSQL context 暴露到 Port，不得把共享 runtime 当业务 SQL hub |
| Endpoints | Contracts、Application use case | Infrastructure service、`PoolAI.Infrastructure.Postgres`、DbContext、Npgsql/SQL、Domain Entity 序列化 |
| Host | 注册扩展和实现程序集；Api/Worker 注册 Host-local PostgreSQL runtime | 业务分支、Service Locator、跨 Host 引用；Migrator 不得注册 runtime application UoW |

Command 与 Query 语义固定如下：

1. R1 控制面短 Command 固定按“认证/资源 Policy → 规范化请求并计算 hash → `IUnitOfWorkFactory.BeginAsync(...)` → Handler 保留 `IUnitOfWork` 并只向参与者传 `unitOfWork.Context` → 插入/锁定幂等行 → 同摘要 completed 则返回原响应；否则校验输入/版本 → 加载 Aggregate/执行 Transaction Script → 同 Context 同步处理 Domain Event → append audit 与版本化 Integration Event outbox → 保存安全响应快照 → commit”执行。`IUnitOfWork` 暴露独立 context 且不实现/继承它；只有 Handler 持有 commit/dispose capability，commit 是一次性终态，未 commit dispose 必须 rollback，终态后 context 失效且不得跨 Command 缓存/复用。任何参与者不得调用独立 `SaveChanges/Commit`，不得先提交 `in_progress` 再执行业务。
2. 已完成幂等记录的同摘要重放在资源版本比较之前返回原响应，但仍需先通过当前主体认证/授权；同 key 不同 hash 冲突。owner/generation/lease 只用于明确登记的异步/长任务或运维接管流程，claim/heartbeat/complete 都必须 CAS 同一随机 owner + 单调 generation，旧 owner 零更新即停止；它不是 R1 控制面短 Command 的第二个预占事务。
3. Query Handler 使用独立无跟踪 query model/DTO，不调用写 Repository、不产生 Domain Event、不修改 checkpoint；不得为了“复用”加载完整 Aggregate。
4. Domain Event 只表达同一 bounded context 内已发生的领域事实，可在 commit 前处理同事务副作用；Integration Event 是版本化 Published Language，必须随业务事实写入 Outbox，commit 后按 at-least-once 发布。
5. Outbox publisher 只 claim/publish/mark，不执行业务规则；durable consumer 使用 `(consumer_name, message_id)` Inbox 或“事实重算 + event sequence checkpoint 同事务”的等价 Idempotent Consumer。毒消息达到上限进入 dead，人工重放创建带 `replay_of` 的新消息，不能伪造 published。
6. `usage_attempts` 是 GroupQuota 的 immutable settlement fact，不是 Usage 的可写 read model。R1 不再另造与 SQL 漂移的 `AttemptUsageRecordedV1/AttemptUsageAdjustedV1` 传输名；Usage 只消费 `topic=poolai.quota.v1`、`schema_version=1` 的冻结 `event_type` union，其中 `settled`、`expired + payload.metadata.conservative_expiry=true` 和 `usage_adjusted` 分别触发 attempt 新增/保守新增/修正投影，并通过 `IAttemptSettlementFactReader` 按 `attempt_id` 重算详细 DTO 和小时桶。其余事件只驱动 quota snapshot/生命周期失效，不增加 usage；精确映射以数据库执行规格第 8 节为准。
7. 同一 Command 的 module Repository 与 Operations Idempotency/Audit/Outbox adapter 在 Infrastructure 内将 vendor-neutral context type-check 为 ADR 0002 的 concrete PostgreSQL capability，并复用同一 connection/transaction。类型不匹配必须在任何写入前 fail-closed；不得 fallback 新连接/事务。共享 runtime 不接收任意 SQL delegate，不拥有业务表 SQL/Repository/Migration，也不以 `TransactionScope`、ambient/`AsyncLocal` 或静态/全局 registry 提供隐式事务。

版本化 Integration Event envelope 固定含 `message_id`（等于 outbox `id`）、`topic`、`event_type`、`schema_version`、outbox 全局 `event_sequence`、可空 `source_event_sequence`、`aggregate_type`、`aggregate_id`、可空 `aggregate_version`、`deduplication_key`、`correlation_id`、可空 `causation_id`、`occurred_at`、对象形态 `payload` 和可空 `replay_of`。Domain Event 只在同 Context 内收集/同步处理，不写通用 outbox；只有映射后的 Integration Event append。消费者忽略未知可选字段；未知 major schema 进入 dead/告警，不能静默跳过后推进 checkpoint。

| 执行备份恢复、迁移、bootstrap Admin | 部署身份 | — | — | — |

补充约束：

1. Operator 可以替换秘密但永远看不到旧值或新值回显。
2. Admin 代用户撤销 Key、重置密码或修改订阅时必须填写原因；审计记录 actor、target、before/after 摘要、request_id 与 idempotency key。
3. 四种角色都可通过 `/api/v1/me/api-keys*` 管理本人 Key；只有 Admin 可通过带目标 `userId` 的管理端点代理列出、创建、查询、修改或撤销他人 Key。代理 POST/PATCH 在 body 中强制 `reason`，DELETE 强制 `X-Change-Reason`。
4. 四种角色都可读取 `/api/v1/me/security-events` 的本人脱敏安全事件摘要；完整审计事件、管理员理由、before/after changes、幂等键摘要、完整 IP 与其他用户信息不得进入该投影。
5. Auditor 的导出使用同一字段脱敏策略；导出不是绕过权限的后门。
6. 数据库运行身份与部署身份分离；RBAC 不授予 SQL/Secret Provider 访问权。

### 3.3 撤销与禁用的一致性

Admin 的 User/API Key/Subscription/Group disable、pause 或 revoke 响应必须在 PostgreSQL canonical 行事务提交后才返回成功。提交后的授权语义为强一致，不采用“最多 60 秒生效”的缓存窗口：

1. 每个新模型请求在进入 Account lease/reservation 前，使用一个有索引的 PostgreSQL 查询强读 API Key、User、canonical Subscription 和 Group 的当前状态、有效期与版本。Redis 只可定位候选/做负向预筛或触发失效，缓存的 active 结果永远不能直接授权。
2. 并发边界以 canonical 查询的数据库快照为线性化点：查询在线性化点早于 revoke 提交的当前 attempt 可以继续并必须结算；线性化点晚于提交的新请求必须拒绝。
3. failover 是新的 attempt，必须再次执行 canonical 强读。原 attempt 准入后发生 revoke 时，不允许借 failover 继续访问另一个 Account。
4. `/v1/models` 和 `/v1/usage` 同样强读 Key/User/Subscription/Group 状态；quota exhausted 不等于授权撤销，仍允许 usage 查询。
5. mutation 提交后发布 Redis invalidation 是性能优化；发布失败不得改变上述语义。缓存恢复或实例滚动不得重新激活已撤销状态。
6. 强读失败返回 503 且不得调用上游。日志记录 canonical 版本和拒绝原因，但不记录 Key HMAC/明文。
7. 每个 JWT 认证的控制面请求在执行 Policy 前强读 PostgreSQL User `status`、`role`、`token_version`；JWT claim 只是候选身份，不是当前授权真相。Redis token-version 快照只能提前拒绝，不能授权。
8. Admin 变更 User 状态或角色时，在同一 PostgreSQL 事务中递增 `token_version`并写审计。提交前已通过强读的当前控制面用例可完成；提交后开始授权的任何请求必须立即拒绝旧 JWT。

## 4. 领域状态机与边界行为

### 4.1 Subscription

一条 canonical Subscription 由 `(user_id, group_id)` 永久唯一标识。Template 只提供名称、说明、默认有效期和 Group，复制后不产生价格或个人配额。

持久化 `status` 只有 `active`、`suspended`、`revoked`。`scheduled` 和 `expired` 只允许作为按 PostgreSQL 当前时间计算的 `effective_status`，不得写入 status，也没有 Worker 到期写回任务：

~~~text
if status == revoked                         => revoked
else if status == suspended                  => suspended
else if db_now < starts_at                   => scheduled
else if db_now >= expires_at                 => expired
else                                         => active
~~~

| 当前持久状态 | 允许操作/事件 | 下一持久状态 | 规则 |
|---|---|---|---|
| 不存在 | 分配，`starts_at < expires_at` | active | 不论当前是否到达 starts_at 都写 active；effective 可为 scheduled/active；创建 canonical 行并审计 `assign` |
| active | extend | active | 只更新 `expires_at`；不得新建行，不改变 Group 配额 |
| active | pause / revoke | suspended / revoked | 必须填写原因；时间经过不改变持久状态 |
| suspended | extend | suspended | 可先延长，但仍不可访问 |
| suspended | resume 且当前位于有效期 | active | 若已过期，必须在同一命令中更新有效期后恢复 |
| suspended | revoke | revoked | 普通 resume 不可撤销 revoke |
| revoked | regrant | active | 仅 Admin 明确操作；复用 canonical 行、设置新起止时间、生成新版本并保留完整审计链 |
| active/suspended/revoked | 时间到达 starts_at/expires_at | 状态不变 | 只改变 effective_status，不写行、不增加 version/审计 |

有效访问要求 User、Group、API Key 都有效，且 canonical Subscription 的 `effective_status=active`。API/查询同时返回持久 `status` 和派生 `effective_status`，UI 和授权只使用后者表达时间状态。所有变更要求 `Idempotency-Key` 和 `If-Match`；同键同请求返回原结果，同键异参返回 409。

### 4.2 API Key

API Key 持久化 `status` 只有 `active`、`disabled`、`revoked`；`expired` 仅是派生 `effective_status`，没有 Worker 写回：

~~~text
if status == revoked                                      => revoked
else if status == disabled                                => disabled
else if expires_at is not null and db_now >= expires_at   => expired
else                                                       => active
~~~

| 当前持久状态 | 事件 | 下一持久状态 | 边界行为 |
|---|---|---|---|
| 不存在 | 用户选择已有有效 Subscription 的 Group 创建 | active | 生成 `sk-pool-` 前缀 Key；明文只返回一次；保存 prefix + HMAC |
| active | disable | disabled | 新模型请求和 `/v1/usage` 都拒绝；历史事实保留 |
| disabled | enable | active | 仅在 DB 时间未过 expires_at 且当前有有效 Subscription 时允许；否则 409 |
| active/disabled | revoke | revoked | 终态，HMAC 永不复用 |
| active/disabled/revoked | 时间到达 `expires_at` | 状态不变 | 只改变 active Key 的 effective_status；到期时间可为空，表示凭据无固定到期日而非配额无限 |
| 任意 | change group | 拒绝 | 创建新 Key 并撤销旧 Key |
| 任意 | rotate | 新 Key active，旧 Key revoked | 不是原行改密；两个操作在一个事务/用例内审计 |

API Key 只从 `Authorization: Bearer` 读取。模型 API 和 `/v1/usage` 不接受 query、Cookie 或自定义 URL 参数中的 Key。列表只返回 ID、name、prefix、group、持久 status、effective_status、到期时间和最近使用时间。

### 4.3 Group

| 当前状态 | 事件 | 下一状态 | 进入条件/效果 |
|---|---|---|---|
| 不存在 | create | disabled | 默认不接流量；创建操作不隐式创建无限 quota |
| disabled | activate | active | 已有 current quota period、总量大于 0；Orchestrator 取得 ready Supply Configuration 的 opaque versioned evidence，GroupQuota 在自身 UoW 原子记录 token/observed_at；数据库只读 guard 再验证至少一个启用的 R1 模型映射和可调度 Account |
| active | disable | disabled | 立即停止新模型请求；不得释放或丢弃既有 reservation；结算继续 |
| disabled | archive | archived | 无 pending reservation；无 `effective_status` 为 active/scheduled 的 Subscription；只读保留历史 |
| archived | reactivate/delete | 拒绝 | R1.1 终态且不物理删除 |

R1.1 Group 内 Account 故障切换允许，跨 Group fallback 禁止。调度查询、数据库约束/检查任务与验收测试必须共同保证 `routing_group_id = quota_group_id`。

### 4.4 Account、Channel 与供应健康

Account 和 Channel 的持久 `status` 均只有 `active`、`disabled`、`retired`；新建默认 `disabled`，`retired` 是终态且不物理删除。两者创建时必须显式选择不可变 `provider=openai/openai_compatible`；公开 `platform=openai` 只是 R1 入站协议族。Account 健康与持久状态分离，只有 `unknown`、`healthy`、`degraded`、`cooling`、`unhealthy`；健康变化不得暗中改写 persistent status。

| 资源/当前状态 | 事件 | 下一状态 | 调度与数据效果 |
|---|---|---|---|
| Account/Channel 不存在 | create | disabled | 不接流量；Account credential 只可写入/替换不可回读 |
| disabled | activate | active | Channel 需至少一个 R1 模型映射；Account 需 credential/Base URL 验证通过且并发上限大于 0 |
| active | disable | disabled | 提交后的新 attempt/failover 强读并拒绝；已准入 attempt 可完成且必须结算 |
| active/disabled | retire | retired | Account 必须已无 enabled binding，否则 `account_in_use`；Channel 必须已无 non-null Supply Configuration 引用，否则 `channel_in_use`。前置判定只读 Supply 表，先 PATCH 清理/禁用引用；成功后从新调度/绑定永久排除并保留历史外键 |
| retired | activate/disable/delete | 拒绝 | 如需重新供应必须创建新资源 |

Account 初始健康为 `unknown`。主动探测和被动反馈可以在 healthy/degraded/cooling/unhealthy 间转换；`cooling` 的 `retry_at` 使用 Redis `TIME`，到期后回到 `unknown` 并等待唯一 half-open 探测。新调度只允许 persistent active 且 health=healthy/degraded 的 Account，degraded 降权评分；unknown/cooling/unhealthy 不可调度。Channel 必须 active。Supply 在每个新 attempt/failover 选择候选前强读持久状态；缓存只优化拒绝。

breaker 作用域固定为 Account，运行态 `closed/open/half-open` 与公开健康的映射及判定如下：

| 输入/状态 | breaker 动作 | Account health/调度 |
|---|---|---|
| 新建或冷却刚到期 | half-open；Redis 只允许一个 10s probe owner | unknown，不接普通流量 |
| 2 次连续 probe 成功 | closed | healthy，可调度 |
| closed 窗口内有失败但未达阈值 | 保持 closed | degraded，降权 |
| 30s 窗口最少 10 次且网络/408/5xx/首包超时失败率 `>=50%`，或连续 5 次失败 | open 30s；连续 reopen 按 2 倍增长、最多 300s，并加 0..10% jitter | cooling，不调度 |
| 上游 429 | open；优先使用合法 `Retry-After`，否则 30s，范围 1s..24h | cooling，不调度 |
| 上游 401/403 凭据失败 | 立即 open，不做自动 half-open；需凭据替换或管理员/健康 Worker 验证 | unhealthy，不调度 |
| 请求自身 4xx（除 408/429） | 不计 breaker 失败 | 不改变健康，不因该错误 failover |
| half-open 任一 probe 失败 | 立即 reopen | cooling/unhealthy 按失败分类 |

所有窗口、阈值、open 时长与 probe owner 使用版本化配置和 Redis `TIME`；多实例不能各自维护只在内存可见的 breaker。Redis 状态更新后由 Supply health writer 持久化 canonical health/retry_at；PostgreSQL/Redis 不一致时按更严格状态拒绝。不得把重试次数同时计入多个嵌套 retry policy。

`group_supply_configurations` 是以 `group_id` 为稳定身份的 Supply Aggregate，拥有可空 configured `channel_id`、独立单调 version/ETag 与 `group_accounts` 子项；R1 提供独立 GET/POST/PATCH 资源而不提供 DELETE。Group create/update DTO 不包含 Channel 或 binding。Supply Configuration POST 使用自己的 `Idempotency-Key`，PATCH 使用自己的 `Idempotency-Key + If-Match + reason`；Group ETag 不能授权 Supply 写，Supply mutation 也不推进 Group version。多行绑定替换可使 Supply version 跳号，客户端只能做强 ETag 相等比较。每个 configured Channel/binding mutation command 必须先以 expected Supply version `FOR UPDATE` 锁 Configuration，再写 Aggregate 子项；Supply Repository 不暴露可绕过该预锁的 binding 写方法，并以双连接测试证明并发替换/启停不死锁、不丢 version。

Account/Channel retire command 不读取 GroupQuota：Account 仍存在任一 enabled binding 时返回 `account_in_use`；Channel 仍被任一 `channel_id IS NOT NULL` 的 Supply Configuration 引用时返回 `channel_in_use`。管理员必须先 PATCH Supply Configuration 清理/禁用引用再退役；不得根据 Group 是 active、disabled 或 archived 改变结果，也不得为退役新增跨 Context Orchestrator。

仅 disabled/active 且未 retired 的 Account 可绑定；解绑只影响新 attempt，不改写旧 reservation/usage。`Application.Orchestration` 的 Group activation use case 先通过 `IGroupSupplyReadiness` 取得覆盖配置、Channel、eligible Account/binding version、lifecycle/health 与数据库观察时间的 opaque `v1.` readiness token，再调用 GroupQuota 仅以 expected Group version 做 CAS；GroupQuota 在自己的单一 UoW 原子记录 token/observed_at、Group 状态、幂等响应、审计与 outbox，但不解析 token 或把它当 lease/capability。激活时至少需要一个 active Channel 映射和一个 active + healthy/degraded 的已绑定 Account；数据库 guard 保留为只读防绕过。Orchestrator 不建立跨 Context 事务；激活提交后 Supply 失效不回滚、自动禁用或改写 Group/evidence/version，新 attempt/failover 仍按当前 Supply Configuration canonical 强读拒绝并告警。

### 4.5 Group quota period

| 当前状态 | 操作 | 下一状态 | 原子结果 |
|---|---|---|---|
| 不存在 | initialize(total) | current | 创建 quota stable row、current period 和 initialize event |
| current | adjust-total(new total) | current | 保留 consumed/reserved；更新 total/version；写 adjust event |
| current | manual-reset(new total) | closed + 新 current | 关闭旧周期、创建新周期、切换 `current_period_id`、写 reset event，同一事务提交 |
| closed | adjust/reopen | 拒绝 | 只允许迟到 reservation 结算和由其产生的 adjustment |

R1.1 不接受 `reset_at`、timezone reset policy 或自动 reset 配置。旧周期的 pending reservation 永远绑定旧周期；重置不迁移、不释放它们。迟到实际用量仍写旧周期并可形成 overage。

配额派生状态（其中 `reserved-full` 只供内部判定/观测，不持久化也不进入公开 Schema；公开 quota status 始终只有 `active/exhausted/disabled`）：

- `exhausted`：`consumed_tokens >= total_tokens`，新模型请求返回 `429 group_quota_exhausted`；
- `reserved-full`（内部）：`consumed_tokens < total_tokens` 且 `consumed + reserved >= total`，持久化 period 仍为 current，对外 status 仍为 `active`；当前 attempt 预留返回 `429 group_quota_reserved` 和 `Retry-After: 1`；
- `active`：`consumed + reserved < total`；
- Group disabled 时先返回 `403 group_disabled`，不泄露精确剩余。

每次 reserve 在锁定当前 period 后必须按以下顺序判定，不得只看 `remaining_tokens` 选一个模糊 429：

1. `consumed >= total` → `429 group_quota_exhausted`，无 `Retry-After`；
2. `estimate > total - consumed` → `429 group_quota_insufficient`，`retryable=false`、无 `Retry-After`；该请求即使所有在途预留释放也无法容纳；
3. `estimate > total - consumed - reserved` → `429 group_quota_reserved`，`retryable=true`、`Retry-After: 1`；
4. 否则原子增加 reserved 并成功。

`total`/`estimate` 输入是安全整数 JSON number，但 consumed/reserved/actual/adjustment/overage 及所有 Group/Account 累计事实在 PostgreSQL 中是 `numeric(78,0)`、在 .NET 中是 `BigInteger`。对外 `AggregateTokenCount` 是匹配 `^(0|[1-9][0-9]{0,77})$` 的十进制 string；不允许科学计数法、符号、前导零或截断。78 位溢出必须使整个结算/调整事务失败、保留 reservation 可恢复状态并发出 P0 告警。

### 4.6 Reservation

| 当前状态 | 事件 | 下一状态 | 计数变化 |
|---|---|---|---|
| 不存在 | reserve 成功 | pending | period `reserved += estimate` |
| pending + dispatch=`not_started` | `mark-dispatch-started` | pending + dispatch=`started` | 与 dispatch event/outbox 同事务；提交后才允许 Adapter 发送任何上游请求字节 |
| pending + dispatch=`started` | 上游有可靠 usage | settled | `reserved -= estimate`，`consumed += actual` |
| pending + dispatch=`started` | 上游已调用但无 usage/结果不确定 | settled | 按持久化的 estimated input/output 保守核销并标记 estimated |
| pending + dispatch=`not_started` | 确认未向上游发送 | released | `reserved -= estimate`，`consumed` 不变 |
| pending + dispatch=`not_started` | lease 到期且回收器 CAS 成功 | expired | `reserved -= estimate`，`consumed` 不变，写 expire event |
| pending + dispatch=`started` | lease 到期且回收器 CAS 成功 | expired + estimated fact | `reserved -= estimate`、`consumed += estimate`，写 immutable attempt/event/outbox；迟到真实 usage 走 adjustment |
| settled | 后得真实 usage | settled | 追加 adjustment，`consumed += actual - prior_actual`；不得把 consumed 调成负数 |
| expired + dispatch=`started` + estimated fact | 迟到可靠 usage/零执行证据 | 保持 expired | 追加唯一 adjustment/event 并更新 `adjusted_at`，以原 estimated fact 为 previous 修正原 period；不能改回 settled 或清除终态证据 |
| released 或 expired + dispatch=`not_started` | 声称有迟到上游 usage | 拒绝并告警 | 该终态证明未越过 dispatch fence，不得追记 attempt/adjustment 或改变 consumed；若出现证据则视为 P0 不变量破坏 |

`attempt_id` 全局唯一，reservation 对 `(period_id, attempt_id)` 唯一；usage attempt 对 `attempt_id` 唯一，同请求另有 `(request_id, attempt_index)` 唯一。reservation 持久化 `estimated_input_tokens + estimated_output_tokens = estimated_tokens`，供 dispatch 后崩溃的保守核销使用。每个状态转换使用 compare-and-set，重复 mark-dispatch/finalize/release/expire 不重复改变计数；`started` 状态永远不能调用零消耗 release。

Group quota reservation lease 参数固定为：非流式 300 秒、每 60 秒续租、从首次调用上游起最大总时长 600 秒；流式滚动 120 秒、每 30 秒续租；单流最大 7,200 秒；客户端断开后内部 drain/结算最多 15 秒；回收器每 30 秒扫描。非流 600 秒到期必须取消上游、进入有界 drain 并用已知 usage 或保守估算结算，不得让活跃非流请求被 sweeper 误回收。reservation 截止时间取 PostgreSQL clock；Routing 只有 Redis Account lease 与 RPM，取 Redis `TIME`；R1 无 User lease。

### 4.7 配额管理边界

1. `total_tokens` 和单 attempt `estimate` 输入范围是 `1..2^53-1` JSON number，不接受 string；total 调整必须包含原因、`Idempotency-Key`、当前 version/`If-Match`。
2. 降低总量到 consumed 以下是合法操作，但 UI 必须预警，提交后立即 exhausted；实际 consumed 不截断。
3. 降低总量到 `consumed + reserved` 以下不撤销既有流；停止新预留，后续核销允许 overage。
4. reset 需要二次确认、新总量和原因；它不是“清零历史”，旧 period、事件和用量永久保留。
5. `/v1/usage` 不创建 reservation、不消耗 Token；Group exhausted 时仍可查询，Key/用户/订阅失效或 Group disabled 时按鉴权错误拒绝。
6. PostgreSQL 不可用时所有依赖它的公开路径统一返回 `503 dependency_unavailable`，绝不依据 Redis 快照、旧 Supply snapshot 或 activation evidence 放行。Redis 不可用且无法安全获取分布式 Account lease 时模型请求返回 503；不能绕过硬并发。
7. R1.1 不存在配额赠送、个人补偿或 quota package；纠错只能由 Admin 调整 Group total 或追加有原因的账本 adjustment。
8. OpenAI 单请求 usage 字段继续输出 JS 安全整数 number；入站上下文/输出上限必须保证正常单请求不越界。若上游仍返回超界单请求 usage，先无损持久事实，再返回/stream 终止为 `upstream_usage_out_of_range`，绝不饱和或四舍五入。

### 4.8 Gateway phase、Retry/Failover 与 Bulkhead

一次上游 attempt 的 phase 是持久 dispatch fence 与当前进程输出状态的组合，必须在 `AttemptContext` 内单向推进并进入日志、trace 和测试 fixture。PostgreSQL 只持久 `dispatch_started_at`；逻辑 `dispatch_state` 唯一派生为 `NULL ⇒ not_started`、非空 `⇒ started`，不另存同义列。`downstream_headers_committed/business_output_started` 是活请求的输出边界，不作为崩溃后继续执行的事实；崩溃恢复见到 started 一律按 dispatch 后规则核销：

| phase | 可否换 Account | reservation 处理 | 对客户端 retryable |
|---|:---:|---|---|
| `prepared`：未提交 dispatch fence、未开始任何上游 I/O | 仅本地/准入临时错误可重新选 Account | 可 release；新 Account 是新 attempt | 只对明确临时且未 dispatch 的错误为 true |
| `dispatched_no_downstream_headers`：dispatch=`started`，下游响应头尚未提交、无业务输出 | 只有 Adapter 证明未写出上游字节、上游以 401/403/429 明确拒绝执行，或协议 capability 提供可验证幂等时允许；还须总 attempt/retry budget 未耗尽 | 当前 attempt 必须 settle：可靠未执行证据记实际 0、`usage_source=confirmed_no_execution`、`is_estimated=false`，否则按已知 usage/保守 estimate；不得零消耗 release | 5xx、首包超时、发送后断连默认 `upstream_dispatch_ambiguous` 且不可重试；服务端内部安全 failover 不暴露中间错误 |
| `downstream_headers_committed`：下游响应头已提交、无有效业务事件 | 禁止透明切换或伪造新 HTTP 状态；流式协议只能按契约终止 | settle/adjust | false；不能诱导自动重放同一流 |
| `business_output_started`：已输出任一有效 JSON/SSE 业务字节 | 禁止 | drain 后 settle/保守核销；流错误只作为终止事件或断流 | false；调用方如需新请求必须显式决定 |

401/403 上游凭据拒绝时立即把当前 Account 标为 unhealthy，合法 429 遵守 `Retry-After` 并进 cooling；但状态码与“未提交下游 Header”本身均不授权重放。只有 Adapter/transport 提供可审计的零字节、明确未执行证据，或 operation capability 有受契约测试证明的上游幂等保证时，才可在 `dispatched_no_downstream_headers` 内切换其他同 Group Account，且不能对同一 Account retry；请求自身其他 4xx 不 failover。DNS/TCP/TLS/建连失败也只有 Adapter 能证明未写出请求字节时才可在总截止内 failover；上游 408、5xx、首包 timeout、发送后断连和 stream idle timeout 默认属于执行结果有歧义。所有 POST 都默认视为非幂等，不能直接套用框架默认 HTTP retry；只有 Gateway 的逐 attempt Process Manager 能按本矩阵发起新 attempt。

同一 Api Host 的资源舱壁只允许以下四个独立 concurrency policy，数值与 Redis 运行契约一致，不存在“queue 默认值”的二次解释：

| policy | 路由 | permit | FIFO queue | 拒绝 |
|---|---|---:|---:|---|
| NonStream | 非流 `/v1/responses`、`/v1/chat/completions` 与 `/v1/models` | 200 | 0 | `429 gateway_overloaded` + `Retry-After: 1` |
| SSE | 流式 `/v1/responses`、`/v1/chat/completions` | 600 | 0 | 尚未提交 SSE Header 时返回同上错误 |
| Control | `/api/v1/*` | 100 | 50 | 队列满或等待 budget/deadline 耗尽时返回 ControlPlaneProblem |
| Usage | `/v1/usage` | 100 | 20 | 队列满或等待 budget/deadline 耗尽时返回 GatewayProblem；只允许 `gateway_overloaded`，不得返回 Group quota 耗尽/预留错误 |

四个 policy 共享进程但不共享并发 semaphore 或等待队列；SSE 不同时占用 NonStream permit，Usage 不占用任一数据面 permit。Bulkhead 必须在 canonical read、Account lease 和 reservation 前取得，拒绝时不创建 request/attempt/reservation。容量测试必须在 SSE 饱和时同时验证登录、管理写入和 `/v1/usage` SLO。

Group RPM 只计 `/v1/responses` 和 `/v1/chat/completions` 两个模型 POST；`/v1/models` 只是共用 NonStream 舱壁容量，`/v1/usage` 使用 Usage 舱壁，两者均不计 RPM、不进入 quota reservation，429 只能是 `gateway_overloaded`。

## 5. 配置契约

### 5.1 来源与加载规则

配置优先级从低到高为 `appsettings.json` → `appsettings.{Environment}.json` → 环境变量 → Secret Provider。环境变量使用双下划线，如 `Auth__Jwt__Issuer`。生产环境不得把 secret 写进镜像、仓库、命令行历史、日志或普通配置文件。

`Reload` 取值：`启动` 表示变更后滚动重启；`热` 表示经部署系统下发的 Options 完整快照可原子替换。R1.1 没有数据库动态 settings 或 HTTP settings mutation；secret 一律使用“启动”，不做进程内密钥热换。所有配置在启动时校验，错误使 readiness 失败且进程退出，不允许带默认危险值运行。

M0-E4 必须实现真实 UDP SNTP v4 readiness gate：Api 用上述 Server/Port/Timeout 发起现场探测，并把 UDP socket connect 到已解析的配置 endpoint，只接受该来源的响应；响应必须为 server mode、支持的 version、`LI != 3`、stratum `1..15`（stratum 0/KoD 一律拒绝），receive/transmit timestamp 均非零，且 originate timestamp 与本次请求的 transmit timestamp 字节完全一致。实现按距注入 `TimeProvider.GetUtcNow()` 最近的 NTP era 解开 32-bit seconds 后，再用四时间戳算法计算节点偏差，不能把 2036 wrap 当作负时间。超时、来源不符、无效包、KoD 或偏差绝对值大于 5 秒均使 `/health/ready` 返回失败，但不得影响 `/health/live`。`±5s` 是固定发布安全边界，不提供配置键，也不得用隐藏环境变量、固定零值 probe 或仅测试替身绕过。SNTP 是无认证明文 UDP；生产 `Health:Ntp:Server` 必须指向受信内部时间源，并由网络 ACL 限制客户端与服务端，不能把任意公网响应当成授权时钟。M1-E2 在此已验证的节点时钟基础上实现 JWT `TimeProvider`/30 秒 ClockSkew 与 TOTP ±1 step/同 step 防重放；M1 不另造 NTP 状态或把 JWT/TOTP 逻辑提前塞入 M0。

Envelope v1 JSON 字段固定为 `v`、`alg`、`kid`、`wrapped_dek`、`wrap_nonce`、`wrap_tag`、`ciphertext`、`nonce`、`tag`；全部二进制使用无填充 base64url。AAD 规范字符串为 `poolai|v1|<purpose>|<entity_type>|<entity_id>|<field_name>` 的 UTF-8 bytes，purpose 至少区分 `account-credential`、`email-delivery-secret`、`idempotency-response`、`totp-secret`。解密前必须先验证调用方给出的 purpose/entity/field 与 AAD 完全相同；密文不能在不同用户、Account、字段或用途间复制。每次写入使用新的随机 256-bit DEK、content nonce 和 wrap nonce；nonce 不持久复用。轮换先加入新 KEK 并切 current，再后台以 CAS 重包裹 DEK，最后只有在数据库、备份和灾备演练都不再引用旧 `kid` 后才可移除历史 key。未知 `v/alg/kid`、AAD 不匹配或认证 tag 失败一律 fail-closed 并发安全告警，不尝试明文兼容。

首个 Admin 通过一次性部署命令创建，密码从 stdin/Secret Provider 注入；bootstrap 成功后不可再次使用同一令牌。它不是长期配置键。

共同交互规则：

- 403 显示无权页，404 显示不存在页，不能用空表伪装；前端菜单隐藏不是授权。
- 所有写操作使用前端生成的 UUIDv7 `Idempotency-Key`；版本化资源携带最新 ETag/`If-Match`。
- destructive 操作使用资源名确认；Group total 降低/reset、凭据替换、用户禁用和订阅 revoke 必须填写原因。
- 时间展示用户时区并可查看 UTC 原值；quota/Group/Account 累计 Token 作为 `AggregateTokenCount` 十进制 string 用任意精度整数库格式化，不先转 JavaScript `number`；不使用货币符号或“价值”。
- Empty、Error、Loading、Forbidden、Disabled 和 Stale 必须有组件测试，不允许只实现 happy path。

### 6.2 `/key-usage` 安全规则

1. API Key 只存在于受控输入组件和一次请求的内存变量，请求发出后立即清空输入与变量；不写 localStorage、sessionStorage、IndexedDB、Cookie、URL、路由 state、Pinia 持久化或 Service Worker cache。
2. 只用 `Authorization: Bearer <key>` 调用同源 `GET /v1/usage`；禁止 query key、表单 GET、第三方代理或浏览器直连上游。
3. 输入使用 password 模式，关闭 spellcheck、自动大写和自动完成；不得自动复制、回显完整 Key 或在错误信息显示 Key。
4. 页面设置 `Cache-Control: no-store`、`Referrer-Policy: no-referrer`、严格 CSP；该路由禁用第三方 analytics、session replay、客服脚本和错误上报 breadcrumbs 中的输入/Headers。
5. 前端 HTTP 拦截器、浏览器 console、后端 access log、反向代理和 trace 必须遮蔽 Authorization。测试需检查 DevTools 等价的 request URL、storage、console 和 telemetry payload。
6. 查询结果可以留在当前页面内存，刷新即清除；不得包含 Account、其他用户、其他 Key 或当前用户贡献量。
7. 连续认证失败接受独立的 IP/匿名防暴力限制；响应保持统一，不能通过时延或错误码枚举有效 Key prefix。

### 7.3 Trace 与审计

网关 trace 层级固定为 `gateway.request` → `admission.<bulkhead>` → `auth` → `canonical-access.read` → `model.map` → 每个 `attempt` 下的 `canonical-access.read`、`canonical-supply.read`、`account.select/lease`、`quota.reserve`、`quota.dispatch-fence`、`upstream.call/stream`、`quota.finalize`。attempt span 必须记录低基数 `phase` 与 breaker state；首 attempt 可在不改变线性化点的前提下复用入站访问强读结果，但仍需在调度前记录 Supply 强读；failover 必须有新的访问与 Supply 强读 span。秘密与模型内容不进入 span attribute/event。

审计事件至少包含：event_id(UUIDv7)、event_type、actor id/role、target type/id、reason、before/after 非秘密摘要、request_id、idempotency key hash、source IP、user agent 摘要、DB timestamp。用户/角色、订阅、Group、quota、Channel/Account、凭据替换、API Key、配置、登录/TOTP/refresh 重放、迁移和恢复都必须审计。审计为 append-only；应用角色没有 UPDATE/DELETE 权限。

### 7.4 密码重置 email outbox

- `POST forgot-password` 对存在/不存在邮箱都返回相同 202 语义和近似处理路径，并先经 IP/账户族群限流；不在响应、日志或 metrics 暴露邮箱是否存在。
- 对存在的 active User，Identity 在同一 PostgreSQL 事务内生成一次性 token、只保存 HMAC/hash + pepper version + DB-clock `expires_at`，并插入 email outbox。邮件需要的原 token 和收件地址使用 envelope encryption 保存在 outbox payload；成功或进 dead 终态后清空密文 payload，只留 template/message-id/时间/结果元数据。
- outbox 持久状态为 pending/processing/sent/dead。Worker 用 PostgreSQL session advisory lock 及 owner+generation+lease 的 `FOR UPDATE SKIP LOCKED` 有界 claim，提交 claim 后再调用 SMTP，不在网络 I/O 期间持有事务；失败指数退避 + jitter，最多 `Email:Outbox:MaxAttempts`，超限进 dead 并告警。使用数据库唯一且稳定的 Message-ID，重试只重发同一 token，不生成多个有效 token；sent/dead 后把 recipient/delivery ciphertext 置 null。
- reset 消费使用 PostgreSQL clock 和单次 CAS；成功改密后在同事务中递增 User token_version、撤销 Refresh families 并写审计。过期、已用、错误 token 返回同一对外错误。
- SMTP 是外部依赖；断连不回滚已提交 outbox，但 email lag/dead 必须可观测。任何日志、trace、审计 before/after 或 dead-letter 查询都不得返回原 token、SMTP password 或解密 payload。

### 7.5 通用 Outbox/Inbox

业务模块在当前 UoW 中 append `pending` outbox；Worker 以 owner+generation CAS 转为 `processing`，外部投递成功后转 `published`，临时失败回到 `pending + next_attempt_at`，达到上限转 `dead`。processing lease 到期只允许更高 generation 接管；旧 owner 的 publish/complete 结果必须 CAS 失败。人工重放不改原 dead 消息，而是创建新 message ID、保留相同 correlation、设置 `replay_of` 并记录操作者/原因审计。

Usage projector 在一个 PostgreSQL 事务中执行：插入 `(consumer_name,message_id)` Inbox（冲突即重复）→按 immutable settlement facts 重算受影响小时桶→UPSERT projection→以 owner/generation CAS 推进 event-sequence checkpoint→commit。未知 schema、缺失 sequence 或事实不一致时不推进 checkpoint，消息进入 dead 并告警。Redis Pub/Sub 缓存失效是可重复的 best-effort consumer，可以通过删除操作天然幂等，但仍必须记录发布结果，不能替代 durable Usage projector。

## 8. SLI、SLO 与容量认证

### 8.1 生产 SLO

| SLI | 计算口径 | R1.1 目标 |
|---|---|---|
| 数据面可用性 | 通过语法与鉴权的请求中，平台自身能返回合法网关响应的比例；明确的上游 4xx/5xx 单独计供应失败，平台内部 5xx/超时计 bad | 滚动 30 天 ≥ 99.90% |
| 控制面与 `/v1/usage` 可用性 | 合法请求非平台 5xx 比例 | 滚动 30 天 ≥ 99.90% |
| 网关附加延迟 | 总端到端耗时减上游耗时；不含客户端上传和供应商处理 | 非流式 p95 ≤ 50 ms、p99 ≤ 150 ms |
| 流式首事件附加延迟 | 下游首事件时间减上游首事件时间 | p95 ≤ 100 ms、p99 ≤ 250 ms |
| `/v1/usage` 延迟 | 热数据、合法 Key、服务端耗时 | p95 ≤ 300 ms、p99 ≤ 1 s |
| quota 安全 | 任一 reserve 提交后是否满足提交前 `consumed + reserved + estimate <= total` | 100%；不设错误预算 |
| quota 幂等 | 同 attempt 重放 reserve/finalize/release 后计数是否只变化一次 | 100%；不设错误预算 |
| Token 无损性 | 事实/adjustment/累计是否精确保存并以 `AggregateTokenCount` 输出 | 100%；截断、浮点取整或 78 位溢出任一次即 P0 |
| 用量新鲜度 | 已提交 attempt 到出现在 usage 聚合的时间 | p95 ≤ 30 s、p99 ≤ 60 s |
| reservation 回收 | pending lease 到期到 CAS 回收完成 | p99 ≤ 60 s；最长占用不超过 lease + 60 s（活跃续租流除外） |
| 对账 | 排除仍在聚合 lag 和已知 adjustment 后 `quota.consumed - usage.period.total` | p99 在 60 s 内回到 0；持续 5 分钟非零告警 |
| 密码重置投递 | email outbox 提交到 SMTP 成功接收（排除明确永久地址错误） | p95 ≤ 30 s、p99 ≤ 120 s；dead 持续 1 分钟告警 |
| 授权时钟健康 | API 节点 NTP 偏差与 JWT/TOTP 边界测试 | 偏差绝对值 ≤ 5 s；超限 readiness 失败；JWT skew 固定 30 s，TOTP 仅 ±1 step |
| 安全响应 | Critical 安全告警确认/缓解 | 15 分钟确认，4 小时内缓解或关闭入口 |

错误预算耗尽时暂停非可靠性功能发布；quota 安全、Token 截断/78 位溢出或秘密泄漏出现一次即停止发布并进入 P0 事件响应。

灾备目标：PostgreSQL RPO ≤ 5 分钟、RTO ≤ 60 分钟；完整服务 RTO ≤ 60 分钟。Redis 视为可重建协调层，恢复后旧 lease 按 TTL/数据库 reservation 回收，不能据 Redis 恢复值修改 quota 真相。每季度至少执行一次恢复演练。

### 8.2 参考硬件与认证门槛

参考环境：单个 API 实例 4 vCPU/8 GiB；单个 Worker 2 vCPU/4 GiB；PostgreSQL 4 vCPU/16 GiB、本地 SSD、连接上限按配置；Redis 2 vCPU/4 GiB；服务间 RTT p95 ≤ 1 ms；上游使用可控 mock，固定响应/流量模型。测试数据至少 100 个 Group、1,000 个用户、5,000 个 Key、100 个 Account、1,000 万 attempt 聚合记录。

| 场景 | 负载 | 通过门槛 |
|---|---|---|
| 同 Group 热点预留 | 100 RPS，全部命中同一 Group，非流式，持续 15 分钟 | 无超发、无负计数、无重复核销；平台错误 ≤ 0.1%；满足网关延迟 SLO |
| 混合非流式 | 100 RPS，Responses 70%、Chat 30%，20 个 Group，持续 30 分钟 | CPU p95 < 75%，RSS < 6 GiB，DB pool 等待 p99 < 50 ms，平台错误 ≤ 0.1% |
| SSE 并发 | 单实例 500 条并发流，每秒 1 个 1 KiB event，持续 30 分钟 | 无非预期断流；CPU p95 < 75%，RSS < 6 GiB；首事件附加延迟达标；reservation 持续续租并最终结算 |
| 舱壁隔离 | 占满 data-stream permits 并保持 30 分钟，同时以 20 RPS 登录/控制面和 50 RPS `/v1/usage` | 新数据流快速 `gateway_overloaded`；既有流稳定；控制面与 usage 仍满足各自 SLO，队列/DB pool 无跨舱壁饥饿 |
| breaker 惊群 | 两个 API 实例对同一 Account 注入 60% 5xx、429 与恢复 | 只出现一个 half-open probe owner；open 期间不打上游；恢复满足连续成功阈值后关闭，无 retry storm |
| 长非流 | 100 并发 mock 请求持续 8～11 分钟，回收器每 30s 运行 | 每 60s 续租；600s 前不误回收；600s 取消/drain/结算；无 pending 泄漏或重复核销 |
| `/v1/usage` | 50 RPS，80% 热缓存/20%冷查询，持续 15 分钟 | p95 ≤ 300 ms、p99 ≤ 1 s；不扫原始 attempt 大表；数据 lag p99 ≤ 60 s |
| Worker 积压恢复 | 暂停聚合 5 分钟后恢复，同时保持 50 RPS 网关流量 | 10 分钟内清空积压，不影响 quota 正确性，API p99 不劣化超过 20% |
| Worker 单 owner | 两个 Worker 竞争 sweeper/aggregate/email job，中途中断持锁 PostgreSQL 连接 | 同时仅一 owner；接管时间 p99 ≤ 30s；无重复副作用；Redis 不参与 leader 选举 |
| 故障切换 | 10% attempt 分别注入可证零字节建连失败、capability 证明未执行的 429、无证据首包超时，总 attempt 上限 3 | 只有前两类允许同 Group 新 attempt，每 attempt 独立事实/预留并先 settle；首包歧义不切换；无跨 Group；开始输出后不透明切换 |
| 进程崩溃 | 分别在 reserve 后、dispatch fence 后、上游接受后、首事件后随机 kill API | dispatch 前可零消耗回收；dispatch 后至少按 estimate 计入 consumed 并有 immutable fact，迟到 usage 可 adjustment；无永久泄漏、无低记实际消耗 |

每个场景在相同构建上独立运行 3 次且全部通过。报告必须保存 commit/image digest、配置、硬件、数据规模、脚本、原始指标和结论；只报告平均值不算通过。

| ID | Given | When | Then |
|---|---|---|---|
| AC-001 闭合开户 | 系统仅有 bootstrap Admin | 匿名用户访问 `/register` 或尝试自助创建 | 路由不存在/按契约拒绝；无用户记录；UI 无注册入口 |

| AC-008 canonical Subscription | 已有 user+group Subscription | 用不同 idempotency key 再次分配/延长 | 同一行 version 增加并追加审计；不存在第二条 canonical 行 |

| AC-010 共享 Group 消耗 | 两用户的 Key 均订阅 Group A | 先后完成模型请求 | 同一 period consumed 累加两者全部 attempt；不存在个人 remaining |
| AC-011 原子热点预留 | Group remaining 恰好 100，两个并发 attempt 各 estimate 80 | barrier 同时 reserve | 仅一个成功；另一个 429 `group_quota_reserved` + `Retry-After: 1`；reserved=80 |
| AC-012 永久耗尽 | consumed 等于 total | 调模型并查询 `/v1/usage` | 模型 429 `group_quota_exhausted`；usage 200、`remaining="0"`、can_request=false |
| AC-013 实际超预留 | estimate 100、上游 actual 130 | finalize | reserved 减 100、consumed 加 130，必要时 overage 正确；不截断为 100 |
| AC-014 重复核销 | reservation 已 settled | 重放相同 finalize/outbox | 返回原语义；period、usage attempt 和 event 不重复增加 |
| AC-015 同 Group 故障切换 | Group A 有 Account 1/2，上游 1 失败 | 网关尝试 1 后切 2 成功 | 两条 attempt 事实、各自结算；二者 quota/routing Group 都是 A |
| AC-016 禁止跨 Group | A Account 全不可用，B 有健康 Account | Group A Key 发请求 | 返回无可用 Account；不得选择 B；无 `routing_group_id != quota_group_id` 事实 |
| AC-017 reset 与迟到核销 | 旧 period 有 pending reservation | Admin reset 后旧 attempt 才返回 usage | 新请求只用新 period；迟到 usage 写旧 period；两 period 计数互不污染 |
| AC-018 降低总量 | consumed=100、reserved=20、total=200 | Admin 以正确 version 调整为 90 | 操作成功并强审计；Group exhausted；既有 reservation 可继续结算并形成 overage |
| AC-019 dispatch 后回收与迟到事实 | API 已提交 dispatch fence 后崩溃 | lease 过期且 Worker 以 estimate 保守回收，随后收到可靠真实 usage 或零执行证据 | reservation 保持 expired 终态；以已存在的 estimated fact 为 previous 追加唯一 adjustment/event、更新 adjusted_at 并修正原 period；不能改回 settled 或写新 period；pre-dispatch released/expired 对任何迟到 usage 均拒绝并发 P0 告警 |
| AC-020 DB fail-closed | Redis 有旧 quota/Supply 快照，PostgreSQL 不可用 | 调用模型、控制面或 `/v1/usage` 的数据库依赖路径 | 统一返回 `503 dependency_unavailable`；上游未被调用；无缓存/evidence 放行 |
| AC-021 Redis hard lease | PostgreSQL 可用、Redis 不可用 | 发需要分布式 Account lease 的请求 | 返回 503；不绕过并发；`/v1/usage` 可按 PG 能力继续服务 |
| AC-022 池隐私 | Group A 有多个用户/Account | 任一有效 A Key 查询 usage | 只返回 Group 聚合；请求不接受 group 参数；无用户/Key/Account 明细 |
| AC-023 `/key-usage` 无痕 | 浏览器打开该页并查询 | 检查 URL、全部 storage、console、telemetry、history、Service Worker cache | 都不存在 Key；请求完成后输入/内存变量清空；仅 Authorization header 曾承载 |
| AC-024 六态与权限 | 为每个前端路由准备 loading/empty/error/disabled/forbidden/stale fixture | 执行组件/E2E | 文案、重试、禁用原因和路由守卫符合 6.1，键盘与读屏可用 |
| AC-025 无盈利功能 | 构建 R1.1 工件 | 扫描源码、程序集、OpenAPI、迁移、配置 Schema、路由和前端菜单 | 不含第 1.4 节模块/字段/端点；唯一累计 Token 主体是 Group |
| AC-026 Token 上界 | 管理端或 API 提交 0、负数、`2^53`、小数 | 初始化/调整 quota 或请求 Token 参数 | 422 且错误定位字段；数据库无截断/溢出写入 |
| AC-027 配置启动校验 | 缺少 pepper/signing/envelope secret 或生产使用 `*` CORS | 启动 API | readiness 不通过且进程退出；日志仅指出缺失键，不打印值 |
| AC-028 OpenAI 垂直闭环 | 新用户、Group、Account、订阅、Key 已配置 | 分别调用 Responses/Chat 流式与非流式、models、usage | 契约匹配；Token/attempt/quota/审计闭环；取消与错误均无 reservation 泄漏 |
| AC-029 撤销线性化 | attempt 1 已在 revoke 提交前完成 canonical 强读，Admin 随后 revoke Key/Subscription/User 或 disable Group | revoke 成功后发新请求，且 attempt 1 尝试 failover | 已准入 attempt 1 可完成并结算；新请求强读后拒绝且不调上游；failover 重新强读后拒绝；Redis 仍缓存 active 也不能放行 |
| AC-030 剩余不足分类 | total=100、consumed=40、reserved=0、estimate=70 | reserve | 429 `group_quota_insufficient`、`retryable=false`、无 Retry-After；若 estimate=50 且 reserved=20 则为 `group_quota_reserved` + Retry-After:1 |
| AC-031 大整数无损 | 受控数据使 period/Account 累计超过 `2^53-1` 且接近 78 位 | 结算、adjustment 并查询 quota/Group/Account 累计 | PostgreSQL numeric/.NET BigInteger 精确；API 返回最长 78 位十进制 string；无浮点/截断；超 78 位事务回滚且 P0 告警 |
| AC-032 Supply Configuration 与生命周期 | disabled Group 已有 quota，尚无或有 staged Supply Configuration；Account/Channel 初始 disabled | 独立 POST/PATCH configured Channel/绑定，以 Supply ETag 并发修改后激活；分别在 readiness 观察至提交前、激活提交后 disable/换 Channel/解绑；在 enabled binding/non-null Channel 引用存在与清理后分别 retire | Group DTO 不含 Channel/binding；Group 与 Supply ETag/幂等 scope 独立，Supply mutation 只推进其单调 version且不改 Group。未就绪或提交前竞态由激活/只读 guard 拒绝；成功激活原子记录 opaque token/observed_at；提交后 Supply 失效不自动禁用，当前 attempt 结算，新 attempt/failover 强读当前 Configuration 拒绝；有 Supply 引用时分别 `account_in_use/channel_in_use`，PATCH 清理后可退役且判定不查询 Group status；只有 Supply 写 configuration/bindings |
| AC-033 控制面 JWT 撤权 | Operator 持有有效 JWT | Admin 将其改为 Auditor 或 disable User，随后用旧 JWT 写控制面 | 变更与 token_version 递增同事务；旧 JWT 在下一请求 PostgreSQL 强读后立即拒绝；Redis 旧快照不授权 |
| AC-034 JWT/TOTP 时钟 | 可控 TimeProvider 和 TOTP step | 测 JWT exp 前后 30s skew、TOTP ±1 step、重放已接受 step，并注入 NTP 偏差>5s | 边界符合 DEC-026；同 step 第二次拒绝；NTP 超标 readiness 失败 |
| AC-035 密码重置邮件 | 存在和不存在的邮箱各一，SMTP 前两次失败 | 请求 forgot-password 并让 Worker 重试 | 对外均 202；仅存在用户有一个 token/outbox；稳定 Message-ID、第三次送达；token 只可消费一次；DB/日志/dead-letter 无明文 token/SMTP secret |
| AC-036 非流续租与截止 | mock 上游非流请求运行超过 300s | 观察每 60s 续租并运行到 600s | 活跃 reservation 不被 sweeper 误回收；600s 取消上游、有界 drain 并结算/保守估算；无 pending 泄漏 |
| AC-037 Worker 单 owner | 两个 Worker 竞争同一 job，Redis 可用或不可用 | 获取 PostgreSQL session advisory lock 并中断 owner 连接 | 同时只一个 owner；新 owner 接管；数据库幂等无重复；Redis 中无 Worker leader/fencing key |
| AC-038 压减后的运行面 | 构建 R1 工件并启动 UI/API/Redis | 尝试修改 settings 并扫描 lease key/路由 | settings 页只读且无 mutation endpoint；无 User lease/key/config；仅 Account lease 作容量控制 |
| AC-039 dispatch 崩溃补偿 | attempt 已 reserve，分别在 dispatch fence 前后 kill API | 运行 sweeper 后检查 period/fact | fence 前只释放 reserved；fence 后 `consumed += estimate` 并落 conservative immutable fact；真实 usage 迟到只 adjustment 差值 |
| AC-040 UoW 提交点 | 控制面 Command 同时改变业务事实、audit、idempotency response、outbox | 在每个写入点及 commit 前后注入故障并重放 | commit 前全部回滚；commit 后同键返回原响应；不存在“业务成功但 audit/outbox/响应缺失”或反向记录 |
| AC-041 Outbox poison/replay | 某 Integration Event consumer 持续失败且两个 Worker 竞争 claim | 超过 max attempts 后由 Admin 安全重放 | 原消息 dead 且不再自动 claim；新消息有 `replay_of`/审计；Inbox 防重复，旧 owner/generation 不能 complete |
| AC-042 breaker 半开 | 两 API 实例共享一个故障 Account，先触发 5xx/429 后恢复 | 等待 cooldown 并并发调度 | open 时不调用；全局仅一个 half-open probe；满足 2 次成功后 healthy，失败立即 reopen；401/403 不自动 probe |
| AC-043 Bulkhead 隔离 | 分别占满 NonStream 200/0、SSE 600/0、Control 100/50、Usage 100/20 | 每次同时调用其他三个分区并继续压入已满分区 | 只有已满分区返回 429 `gateway_overloaded` + `Retry-After: 1`，其他舱壁仍达 SLO；SSE 不占 NonStream permit；Usage 过载契约不含 quota 错误；所有拒绝请求无 request/reservation/outbox |
| AC-044 Envelope v1 | 准备 current/old key、篡改 tag、复制到不同 Account/字段、未知 v/kid 和备份恢复 | 解密、轮换、重包裹并恢复 | 仅正确 purpose/entity/field 可解；旧 key 在保留期可解；篡改/错用途/未知版本 fail-closed；日志无 key/plaintext |
| AC-045 事件与错误 Schema | 准备 `GroupQuotaEventV1` 全部 event_type、重复/乱序/未知 major、Envelope/payload 不一致、conservative/pre-dispatch expired，以及控制面/Gateway 两类错误 fixture | 投影并执行 OpenAPI 正反验证 | 只有 settled/conservative expired/usage_adjusted 重算 usage；checkpoint 只在同事务成功后推进；未知/不一致进 dead 且 P0；控制面不能含 OpenAI `error`，Gateway 必须含；models/usage 的 429 只能为 `gateway_overloaded` |

## 12. 全局 Definition of Done 与 CI/CD 质量门

### 12.1 Story/Epic 全局 DoD

每个 Story 只有同时满足以下条件才可完成：

1. 需求关联至少一个 DEC、OpenAPI/数据契约或验收场景；行为改变同步契约和文档。
2. 代码符合项目依赖方向、nullable 与 analyzer 规则；无警告压制，例外有注释和 issue。
3. happy path、边界、拒绝、幂等、并发/取消（适用时）均有自动测试；测试不会依赖执行顺序或墙钟等待。
4. 涉及数据变更提供 forward migration、索引/锁影响说明、空库与升级测试；生产不依赖自动 `Database.Migrate()`。
5. 涉及公开 API 更新 OpenAPI、错误目录、fixtures 和兼容性说明；未定义错误不得外泄。
6. 涉及写操作具备授权、审计、idempotency、乐观并发和秘密脱敏（适用时）；业务事实、audit、idempotency response 与 outbox 在同一显式 UoW 中提交。
7. 涉及后台/异步处理具备 inbox/checkpoint 幂等、owner/generation claim、重试上限、死信/可观测性和创建新消息的安全重放路径。
8. 日志、指标、trace、health 与告警足以定位故障，且不含秘密/模型内容。
9. 前端具备 Loading/Empty/Error/Disabled/Forbidden/Stale 状态、键盘操作、可访问名称和响应式验证。
10. 威胁模型/隐私检查完成；没有引入商业化模型、个人 quota 或跨 Group fallback。
11. Review 已解决；构建产物可复现；变更记录与运行手册已更新。

覆盖率基线：Domain/Application 行覆盖率 ≥ 85%，分支覆盖率 ≥ 75%；整体行覆盖率 ≥ 75%；本次修改行覆盖率 ≥ 90%。覆盖率不能替代契约、并发、故障或安全测试。quota/auth/secret 关键路径若覆盖率下降即阻断。

### 12.2 PR 质量门

按顺序执行，任何一步失败停止合并：

1. 锁定依赖 restore、格式检查、nullable/analyzer、warnings-as-errors、architecture tests；
2. 后端 unit/contract tests，前端 lint/typecheck/unit/component tests；
3. OpenAPI lint、breaking-change 检查、错误码唯一性、JSON/SSE fixtures；
4. Testcontainers PostgreSQL/Redis integration、空库 migration、quota/dispatch concurrency、outbox/inbox/email claim 接管与崩溃恢复；
5. E2E API 主路径与 RBAC 否定测试；前端关键路由浏览器 smoke；
6. secret scan、SAST、依赖/许可证扫描；Critical/High 未豁免项为 0，豁免需到期日与风险批准；
7. 生成 SBOM，构建非 root、只读根文件系统兼容的容器并执行镜像扫描；
8. 对源码、程序集、OpenAPI、DDL、配置和前端执行商业功能/个人 quota/跨 Group 规则否定扫描。

### 12.3 Main/Release 质量门

| 阶段 | 必须通过 |
|---|---|
| Main | PR 全部门；构建不可变镜像；从空库部署临时环境；全部 API E2E、Web 主流程、migration checksum；生成并归档 SBOM |
| Staging | 与生产同拓扑部署；合成 OpenAI 流式/非流式、quota/usage、告警；故障注入；备份与一次恢复 smoke；24 小时 soak 无 quota/secret/跨 Group 异常 |
| Release Candidate | AC-001..045；完整安全测试；第 8.2 节三次负载；RPO/RTO 演练；OpenAPI compatibility；运维/回滚评审 |

| Post-release | 观察 60 分钟 quota rejection、reservation、5xx、usage lag、reconciliation、SSE；异常按 runbook 回滚应用，数据库采用向前修复/恢复方案 |

镜像以 digest 晋级，不在各环境重新构建。数据库迁移默认为 forward-only；回滚应用前必须验证旧版本与新 Schema 的向后兼容窗口。任何需要破坏性 Schema 删除的变更不进入 R1.1 发布迁移。
